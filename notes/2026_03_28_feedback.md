## 2. Output length conditioning

**Deborah feedback:** "which is appropriate because our primary behavioral outcomes and benchmarks target item selection rather than stopping behavior." -> normally before meeting you I would think that people stop because they can't remember any more items. This sentence isn't really clear to me still. I'm not sure what is the difference between the two options - isn't output length the same as 'how many'? "Conditioning on observed output length isolates the model's predictions about which items are recalled from predictions about how many"

**Manuscript locus:** `index.qmd` L310

**Our assessment:** Deborah is pressing on a real modelling limitation, not just a prose problem. The current forced-output-length setup disconnects stopping from memory state, even though in free recall stopping and retrieval structure are plausibly linked. A stronger redesign is feasible because the codebase already supports explicit termination policies. At the same time, exact sequence-likelihood fitting is not currently justified, because the available TalmiEEG recall vectors appear to be sorted recalled positions rather than true production order. That points toward a termination-aware set-likelihood redesign rather than a full observed-sequence likelihood.

**Response options:**
1. `Keep forced output length`: clarify the distinction between identities and counts, but retain externally forced stopping.
2. `Termination-aware set likelihood`: replace forced output length with a stopping model that is part of the latent recall process, starting with `SupportRatioTermination`.
3. `Full sequence likelihood later`: pursue exact observed-sequence fitting only if true recall order can be validated or recovered from source data.

**Current top recommendation:** Adopt a `termination-aware set likelihood` with `SupportRatioTermination` as the current preferred stop mechanism, explicitly replacing forced output length as the preferred path.

**Revision scope:** Model redesign + manuscript reframing.

**Open dependency:** `RetrievalDependentTermination` remains a plausible secondary alternative. Exact sequence-likelihood fitting remains contingent on verifying true recall order, because the stored recall vectors are currently consistent with sorted recalled positions rather than observed production order.
